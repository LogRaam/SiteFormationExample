Opérationnaliser un toggle avec le code et LaunchDarkly

De la création du flag dans la console LaunchDarkly à son évaluation dans le code, voici le parcours technique complet. Les exemples ci-dessous sont génériques et illustratifs.

Vue d'ensemble : le SDK dans votre application interroge LaunchDarkly pour connaître la valeur d'un flag pour un contexte donné (un membre, un appareil…). Le SDK met les règles en cache et reçoit les changements en quasi temps réel. Vous fournissez toujours une valeur par défaut au cas où le service serait injoignable.

1. Créer le flag dans LaunchDarkly

  1. Définir la clé du flag

    Une clé stable, en kebab-case, descriptive et préfixée par l'équipe/domaine : virements-nouvelle-page. La clé ne doit plus changer.

  2. Choisir le type de variation

    Booléen (on/off) pour un release simple, ou multivarié (chaîne, nombre, JSON) pour des variantes.

  3. Renseigner le contexte de gouvernance

    Propriétaire, description, type de switch, date de retrait visée, et catégorie au catalogue (voir Chez ExampleCo).

  4. Définir la valeur par défaut côté serveur

    Par défaut « off » (comportement existant) tant que la nouvelle fonctionnalité n'est pas prête.

  5. Configurer les règles de ciblage

    Cibler des membres, des segments (bêta, interne) ou un pourcentage progressif.

Note : Cette section doit être complétée avec les captures d'écran réelles de la console LaunchDarkly utilisée chez ExampleCo, les conventions de nommage internes et le gabarit de fiche de toggle du catalogue central.

2. Évaluer le flag dans le code

Selon votre stack :

import * as LaunchDarkly from '@launchdarkly/node-server-sdk';

const client = LaunchDarkly.init(process.env.LD_SDK_KEY);
await client.waitForInitialization();

const contexte = { kind: 'user', key: membre.id, custom: { segment: 'beta' } };

// 3e argument = valeur par défaut SÛRE si LD est injoignable
const actif = await client.variation('virements-nouvelle-page', contexte, false);

if (actif) {
  afficherNouvellePage();
} else {
  afficherAnciennePage();
}
LDConfig config = new LDConfig.Builder().build();
LDClient client = new LDClient(System.getenv("LD_SDK_KEY"), config);

LDContext contexte = LDContext.builder(membre.getId())
        .set("segment", "beta")
        .build();

boolean actif = client.boolVariation("virements-nouvelle-page", contexte, false);

if (actif) {
    afficherNouvellePage();
} else {
    afficherAnciennePage();
}
var config = Configuration.Default(Environment.GetEnvironmentVariable("LD_SDK_KEY"));
var client = new LdClient(config);

var contexte = Context.Builder(membre.Id)
    .Set("segment", "beta")
    .Build();

bool actif = client.BoolVariation("virements-nouvelle-page", contexte, false);

if (actif) AfficherNouvellePage();
else       AfficherAnciennePage();
import ldclient
from ldclient.config import Config
from ldclient.context import Context

ldclient.set_config(Config(os.environ["LD_SDK_KEY"]))
client = ldclient.get()

contexte = Context.builder(membre.id).set("segment", "beta").build()

actif = client.variation("virements-nouvelle-page", contexte, False)

if actif:
    afficher_nouvelle_page()
else:
    afficher_ancienne_page()

3. Patrons recommandés

✅ Centraliser l'évaluation

Encapsulez l'appel au SDK derrière une petite fonction/abstraction. Le reste du code ne dépend pas directement de LaunchDarkly et reste testable.

✅ Valeur par défaut = état sûr

La valeur par défaut doit toujours correspondre au comportement existant et stable.

✅ Contexte cohérent

Utilisez une clé de contexte stable (id de membre) pour un bucketing reproductible lors d'un rollout progressif.

✅ Tester les deux états

Vos tests automatisés doivent couvrir on ET off — voir Toggle Ready.

À éviter : appeler le SDK profondément dans des boucles critiques sans mise en cache, coder en dur la clé du flag à plusieurs endroits, ou laisser une branche du if non testée.
