Être « Toggle Ready »

Être « Toggle Ready », c'est intégrer le toggle proprement dans son flux de développement : de la convention de nommage au retrait planifié, en passant par le gating et les tests des deux états.

La liste de vérification

Avant de fusionner du code derrière un flag, passez cette liste :

0 / 9 — commencez la vérification.

Gating : où placer le point de décision ?

Le gating est l'endroit du code où l'on évalue le flag pour bifurquer. Bien placé, il garde le code lisible et testable.

✅ Au plus haut niveau utile

Décidez tôt (ex. au point d'entrée d'un parcours) et passez le résultat, plutôt que de re-tester le flag partout.

✅ Un seul point de vérité

Une fonction fonctionnaliteActive() unique par flag, mockable en test.

❌ Évaluations dispersées

Tester le même flag à dix endroits multiplie les risques d'incohérence.

❌ Imbrication profonde

Des if de flags imbriqués rendent les états ingérables.

Nommage : un exemple de convention

<domaine>-<fonctionnalité>[-<variante>]

✔ virements-nouvelle-page
✔ paiements-3ds-experimentation
✔ ops-integration-partenaire-x-killswitch
✘ test1            (non descriptif)
✘ NouvellePage     (casse incohérente)
✘ flagTemp         (sans domaine ni intention)
Note : Cette section doit être complétée avec la convention de nommage officielle de ExampleCo et les règles du catalogue central (préfixes par équipe, métadonnées obligatoires).

Tester les deux états

describe('page de virements', () => {
  it('affiche la nouvelle page quand le flag est ON', () => {
    const rendu = render({ flag: { 'virements-nouvelle-page': true } });
    expect(rendu).toContain('Nouvelle page');
  });

  it('affiche la page actuelle quand le flag est OFF', () => {
    const rendu = render({ flag: { 'virements-nouvelle-page': false } });
    expect(rendu).toContain('Page actuelle');
  });
});

Anti-patrons fréquents

Le flag « pour toujours »
Un release toggle qu'on ne retire jamais devient de la dette. Fixez une date de retrait et tenez-la.
Le flag comme contrôle d'accès
Un toggle n'est pas un système d'autorisation sécurisé ; ne l'utilisez pas pour protéger des données sensibles.
Les flags couplés implicitement
Deux flags qui doivent toujours être cohérents entre eux sont un piège. Documentez ou fusionnez la logique.
Le chemin OFF non testé
Si seul l'état ON est testé, désactiver le flag en production devient un saut dans l'inconnu.
